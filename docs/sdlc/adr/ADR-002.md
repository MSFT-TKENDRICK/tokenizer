# ADR-002 — Persist tokenizer canvas input per Copilot session, outside the repo tree

- **Status:** accepted
- **Date:** 2026-06-12
- **Affects requirements:** FR-009
- **From facets:** MSF-006
- **Spec:** 002 — Tokenizer Copilot Desktop Canvas

## Context

FR-009 originally read *"The system SHOULD persist the user's latest input text and
restore it when the canvas is reopened"* and carried a `[NEEDS CLARIFICATION: persist
per session/workspace or per user/global?]` marker (interview item AMB-001, BLOCKING).
The scope of persistence is observable behavior — it changes what a user sees when they
reopen the canvas — so the decision is recorded as an ADR, not only a facet.

Relevant facts about the runtime (see the canvas contract facets MSF-007/MSF-010):

- The extension is a Copilot **desktop canvas** wired through `joinSession({ canvases:
  [createCanvas({...})] })`; the SDK exposes a **per-session** context (`sessionId`) and
  a session object used for logging (`session.log`).
- The extension has its own artifacts directory under `$COPILOT_HOME`. Writing user
  text into the **repository working tree** would be wrong: it would pollute the repo,
  risk committing user content, and is not session-isolated.
- A single user can run multiple concurrent Copilot sessions; bleeding one session's
  prompt into another would be surprising and a minor privacy leak.

The alternatives considered were: (a) **per user / global** (one latest-text blob shared
across all sessions), (b) **per workspace**, and (c) **per Copilot session**.

## Decision

Persist the latest input text **scoped to the Copilot session**, keyed by `sessionId`,
in the extension's own **session-scoped artifacts directory under `$COPILOT_HOME`**,
deliberately **outside the repository working tree**. On canvas open within the same
session the persisted text is restored into the input; opening the canvas in a different
session MUST NOT surface another session's text. FR-009 is also strengthened from
*SHOULD* to *MUST* because the chosen mechanism is cheap and deterministic.

Global/per-user persistence (alternative a) was rejected because it cross-contaminates
concurrent sessions and complicates the privacy story; per-workspace (b) was rejected
because the canvas is session-oriented and has no first-class workspace handle.

## Consequences

- FR-009 now has measurable, testable acceptance criteria (see MSF-006), unblocking its
  BDD scenario and TDD loop entry.
- The `[NEEDS CLARIFICATION]` marker is removed from `spec.md`; the BDD feature/eval/TDD
  entry for FR-009 were regenerated and the squad re-grounded (still ✅ GROUNDED, 13/13).
- No user content is ever written into the repo working tree, keeping `git status` clean
  and avoiding accidental commits of prompt text.
- A future change to cross-session ("recent prompts") history would supersede this ADR
  rather than edit it.
