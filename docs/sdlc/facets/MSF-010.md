# MSF-010 — Security targets for the tokenizer canvas (grounded in NFR-003)

> Micro-spec facet: a focused enrichment of the SpecKit spec, generated for an
> identified ambiguity to give squad members the knowledge they need to execute.

**Grounded in:** NFR-003 (also constrains FR-001 server wiring and FR-008 actions)

## Question (from interview)

AMB-005 (MEDIUM, missing-non-functional): the spec has no standalone security
requirement. What security targets must the implementation meet, or is security out of
scope?

## Resolution

Security is **not a separate new requirement** — it is fully expressed by **NFR-003**
plus the canvas's nature as a local, sandboxed, read-only-of-text tool. This facet
records the explicit, testable security posture so the squad does not invent extra
scope (e.g. auth, accounts) that the spec does not call for:

- **Loopback-only binding.** The per-instance HTTP server (FR-001) binds to
  `127.0.0.1` only; it MUST NOT bind to `0.0.0.0` or any externally routable address,
  so the iframe assets are never reachable off-host.
- **No stdout.** The extension never writes to `process.stdout` (reserved for
  JSON-RPC); all logging goes through `session.log`. Leaking to stdout would corrupt the
  protocol and could spill user text into logs.
- **No secrets, no external network.** The tokenizer processes only the user's pasted
  text locally; it makes no outbound network calls and stores/transmits no credentials.
  Model pricing is static data from `web/models.mjs`.
- **No persisted content in the repo tree.** Per MSF-006/ADR-002, the only persisted
  artifact (last input text) lives under `$COPILOT_HOME`, session-scoped, never in the
  working tree — preventing accidental commit of user content.
- **No authentication is in scope.** The tool is a single-user, on-device canvas; auth
  would be invented scope and is explicitly excluded.

## Acceptance criteria

- [ ] The embedded HTTP server's bound address is `127.0.0.1` (loopback); a test/asserts
      it is not bound to `0.0.0.0` or a non-loopback interface.
- [ ] No bytes are written to `process.stdout` during the extension lifecycle; logging
      uses `session.log` (shared assertion with NFR-003).
- [ ] The extension performs no outbound network requests and references no secrets/env
      credentials in its code path.
- [ ] No user-input content is written under the repository working tree at any point
      (shared assertion with MSF-006).
