# MSF-007 — Acceptance criteria for FR-001: canvas registration + loopback server

> Micro-spec facet: a focused enrichment of the SpecKit spec, generated for an
> identified ambiguity to give squad members the knowledge they need to execute.

**Grounded in:** FR-001 (also informs FR-008, NFR-003)

## Question (from interview)

AMB-002 (HIGH, missing-acceptance-criteria): what are the explicit, observable
pass/fail criteria for FR-001 ("register a GitHub Copilot desktop canvas extension …
discoverable in the side panel and serves its UI from a per-instance loopback HTTP
server")?

## Resolution

FR-001 is satisfied by the canvas-contract shape the top-level orchestrator is
implementing (this facet does not change scope; it enriches with the contract facts the
squad needs):

- Entry point is `.github/extensions/tokenizer/extension.mjs` — **ESM only**, no
  TypeScript, no `package.json` declaring `@github/copilot-sdk`.
- It calls `joinSession({ canvases: [createCanvas({ id, displayName, description,
  actions, open, onClose, inputSchema? })] })` so the canvas is discoverable in the
  app's side panel.
- Each canvas **instance** stands up its **own** `http.Server` bound to
  `127.0.0.1:0` (loopback, OS-assigned port) that serves the `web/` iframe assets; the
  `open` handler returns/points the host at that instance's URL.
- The extension **never writes to stdout** (reserved for JSON-RPC); all logging goes
  through `session.log` (cross-references NFR-003).

> Ownership note: the extension files under `.github/extensions/tokenizer/` are authored
> by the top-level orchestrator and are out of this SDLC run's edit scope. The squad's
> unit tests target the ported pure modules (`web/tokenizer.mjs`, `web/models.mjs`);
> BDD/Playwright scenarios target the served iframe page.

## Acceptance criteria

- [ ] The extension exports an ESM entry at `.github/extensions/tokenizer/extension.mjs`
      that registers exactly one canvas via `createCanvas`/`joinSession` with a stable
      `id` and human-readable `displayName`.
- [ ] Opening the canvas serves the iframe UI from a server bound to `127.0.0.1` on an
      ephemeral port (port `0` → OS-assigned); the bound address is loopback only.
- [ ] Each canvas instance gets its **own** HTTP server (per-instance, not a shared
      singleton); closing the canvas (`onClose`) releases that instance's server.
- [ ] No bytes are written to `process.stdout` during registration or serving; logs go
      through `session.log` (verified jointly with NFR-003).
