# MSF-009 — Acceptance criteria for FR-010: responsive recompute of derived views

> Micro-spec facet: a focused enrichment of the SpecKit spec, generated for an
> identified ambiguity to give squad members the knowledge they need to execute.

**Grounded in:** FR-010 (latency budget defined by NFR-002)

## Question (from interview)

AMB-004 (HIGH, missing-acceptance-criteria): what are the observable pass/fail criteria
for FR-010 ("recompute and update all derived views — tokens, summary metrics,
comparison table — responsively as the user edits the input")?

## Resolution

"Responsively" is made measurable by binding FR-010 to the NFR-002 budget: after any
input change, **all three derived views** (token visualization, summary metrics, and the
model comparison table) recompute from the single source of truth (current input text →
`tokenize` → `summarizeTokens` → per-model cost) and the budget for the full recompute is
**< 50 ms for inputs up to 10,000 tokens** (NFR-002). The views are kept consistent: there
is no stale view after the input settles. The recompute is the single code path that also
triggers the debounced session persistence (MSF-006).

## Acceptance criteria

- [ ] Editing the input updates the token list, the summary metrics (characters, bytes,
      words, lines, tokens) and every comparison-table row without a manual refresh.
- [ ] After an edit settles, all derived views reflect the **same** input — the token
      count shown in the summary equals `tokenize(input).length` and equals the count the
      comparison table uses for cost/usage.
- [ ] A full recompute of all derived views for a 10,000-token input completes in under
      50 ms (measured against the ported `web/tokenizer.mjs` + `web/models.mjs`; shared
      with NFR-002's benchmark).
- [ ] The summary metrics match the ported reference exactly: `summarizeTokens(input)`
      values for a fixed fixture equal the rendered characters/bytes/words/lines/tokens.
